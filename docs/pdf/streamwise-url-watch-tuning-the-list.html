StreamWise URL watch — tuning the list

Beginner guide · data/url-watch-list.json · GitHub Action url-watch.yml

What you are tuning

data/url-watch-list.json is a JSON array. Each entry is one monitored URL with a stable id and a human label.

Rule 1 — start small (less noise)

If you monitor too many URLs, you will get too many GitHub issues.

Beginner recommendation:

  • Start with 10–20 URLs that matter most.
  • Prioritize promos, bundles, and carrier pages you rely on.

Rule 2 — regenerate after big catalog changes

When you add/remove lots of sourceUrl values:

cd /Users/chrisanderson/streamwise
npm run catalog:url-watch:generate-list -- catalog-before.json

Then commit and push the updated data/url-watch-list.json.

Rule 3 — use skipAutomation for bot-hostile pages

Some sites return 403 to cloud IPs. If a URL is blocked or too noisy, add: 

{
  "id": "opt-example",
  "url": "https://example.com/pricing",
  "label": "Example pricing page",
  "skipAutomation": true
}

Then monitor that URL with a consumer tool (like Distill) instead of GitHub Actions.

Rule 4 — dedupe mentally (optional)

If multiple catalog options share the same sourceUrl, you do not need duplicate monitor rows unless you want separate labels in issues.

Rule 5 — verify workflow after edits

  1. GitHub → Actions → URL watch → Run workflow
  2. Confirm green checkmark

Common mistakes

  • Editing JSON locally but forgetting to commit + push to main.
  • Thinking “fingerprint changed” always means “price changed” (it does not).
  • Monitoring huge marketing homepages instead of the smallest stable pricing/help URL.
